## Context

当前项目 ai_superintelligent_agent 是一个多智能体 AI 应用，采用 Spring Boot 3.4.5 + Vue 3 技术栈。项目包含多个 AI Agent (Manus, LoveApp, QuizAssistant, HealthAssistant)，支持多种大模型接入 (DashScope, Ollama)。项目缺乏系统性的技术文档，新成员难以快速理解系统架构。

## Goals / Non-Goals

**Goals:**
- 创建完整的项目技术文档，包括项目架构、技术栈、模块说明
- 整理并文档化所有 REST API 接口
- 绘制项目结构图和组件关系图
- 记录前端路由和页面组件结构
- 整理 AI Agent 能力说明和工具列表

**Non-Goals:**
- 不修改任何代码
- 不添加新的依赖
- 不创建运行时代码（如自动生成 API 文档）
- 不包含部署和运维文档

## Decisions

### 1. 文档格式
- **决定**: 使用 Markdown 格式输出文档
- **理由**: Markdown 便于版本控制、代码审查和协作编辑，是技术文档的事实标准
- **替代方案**: HTML/Confluence - 不便于代码管理

### 2. 文档输出位置
- **决定**: 输出到项目根目录的 `docs/` 文件夹
- **理由**: 集中管理文档，便于查找；不污染源代码目录
- **替代方案**: 嵌入代码注释 - 分散且难以维护

### 3. API 文档详略程度
- **决定**: 提供端点列表、请求参数、响应格式的基本说明
- **理由**: 项目使用 Knife4j/Swagger 已生成交互式 API 文档，补充说明文档作为概览
- **替代方案**: 完全依赖 Swagger - 缺少系统级视角

### 4. 文档维护策略
- **决定**: 文档与代码解耦，作为独立文档存在
- **理由**: 避免文档更新成为代码提交的阻碍
- **替代方案**: 代码即文档 - 需要额外工具支持

## Risks / Trade-offs

- [风险] 文档可能随代码更新而变得过时 → **缓解**: 在 README 中说明文档更新周期，建议 PR 审查时检查相关文档
- [风险] 手动维护文档增加工作量 → **缓解**: 保持文档简洁，只记录关键信息，避免过度细节
- [风险] 图形图表难以用 Markdown 表达 → **缓解**: 使用文本图表（Mermaid 语法）或纯文字描述
